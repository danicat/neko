# Neko v0.2.0: The Semantic Refactoring Specification
*Date: March 20, 2026*
*Status: Finalized Design*

## 1. Executive Summary & Core Philosophy

This document is the exhaustive technical specification for Neko v0.2.0. It consolidates hours of architectural debate, algorithmic refinement, and protocol analysis. The transition to v0.2.0 redefines Neko as a **Semantic Operating System** for AI agents, moving beyond simple file manipulation into deterministic, project-wide refactoring.

### 1.1 Core Mandates
1.  **Pragmatic Full Disclosure**: Statelessness is our primary defense against technical debt. Every edit or creation returns the *Total Reality* of the project's health. We do not prune or hide pre-existing errors; the agent must own the codebase's integrity.
2.  **Stateless Determinism**: Every tool call is an atomic transaction. We rejected long-lived virtual sessions to ensure the disk is always the source of truth for both the agent and the human developer.
3.  **LSP-Authoritative Intelligence**: We eliminate "Neko-side" magic. Syntax validation, semantic linking, and structural outlines are delegated to the Language Server Protocol (LSP).
4.  **Turn-Level Semantic Certainty**: In the MCP architecture, background work is an anti-pattern. Tools must block until the LSP confirms the semantic outcome, ensuring the agent's next thought is based on accurate reality.

### 1.2 Latency Budget & Performance SLAs
To maintain semantic certainty without compromising agent momentum, Neko targets the following p95 latencies:
- **`edit_file`**: 4.0s (includes Formatting, Diagnostic Wait, and RAG Sync).
- **`multi_edit`**: 8.0s (for a standard 5-file batch).
- **`semantic_search`**: 1.5s.
- **`read_file`**: 1.0s (includes "Warm-Start" didOpen).

## 2. Problem Statement & Evolutionary Log

### 2.1 The Baseline Risks (v0.1)
- **Data Integrity**: `edit_file` lacked atomicity, risking file truncation on failure.
- **Semantic Blindness**: Edits were "local-only." Breaking changes in one file would go unnoticed until a manual `build` turn, wasting agent context and conversation turns.
- **Diagnostic Fragility**: Line number extraction used naive heuristics, causing "hallucinated context" in error reports.

### 2.2 Discarded Iterations & Rationale

#### Iteration 1: The "Write-Validate-Move" Gate
- **Proposal**: Write to `.tmp`, run `backend.Validate`, then `os.Rename`.
- **Rejection Reason**: It only solved for *Syntax*. A project can be syntactically valid but semantically broken (e.g., calling a deleted function). It also required maintaining custom AST parsers for every language.

#### Iteration 2: Virtual LSP Simulation
- **Proposal**: Send `didChange` (virtual memory) -> Pull Diagnostics -> If clean, Commit to disk.
- **Rejection Reason**: Over-engineered for an MCP server. It created state-drift between Neko and the filesystem. If the process crashed, the "work" disappeared. It added unnecessary protocol latency for a feature (simulation) that offered little value given our "Direct Write" philosophy.

### 2.3 The Final Path: "Stateless Semantic Navigator"
We chose to embrace **Direct Write + Synchronous Sync**. We trust **Git** as the project's atomic transaction manager and the **LSP** as its semantic oracle.

## 3. Pillar 1: LSP-Aware File Modification (LAFM)

### 3.1 The Hardened `edit_file` Algorithm
Every edit turn must execute this deterministic sequence:

1.  **Tolerant Matcher (The Safety Gate)**:
    - **Normalization**: Strip whitespace, normalize line endings. 
    - **Annotation Shield**: Automatically strip all `<NEKO>...</NEKO>` tags from **both** `old_content` and `new_content` before processing.
    - **Sliding Window**: Levenshtein similarity calculation.
    - **Threshold**: `0.95`.
    - **Failure Logic**: If `score < 0.95`, **abort modification**. Identify Top 3 local maxima (scores > 0.5) and return a ranked Markdown list.
2.  **LSP Pre-Flight (Natural Sync)**:
    - Send `textDocument/didOpen` with the **Current Disk Content**. (Establishing the authoritative baseline).
    - Apply the match in memory to generate the **Proposed Content**.
    - Send `textDocument/didChange` with the **Full Proposed Content**. (Neko uses full-document sync for simplicity and correctness).
3.  **Automated Save Actions**:
    - Request `codeAction` (`source.organizeImports`) and `textDocument/formatting` from the server.
    - Chaining: Apply all `TextEdit`s to the in-memory buffer before the write.
4.  **Direct Write**:
    - `os.WriteFile` the final, clean, formatted content directly to the target path.
5.  **LSP Sync Trigger (`didSave`)**:
    - Fire `textDocument/didSave` (with the `text` field) to force the LSP to re-index and confirm convergence.
6.  **Synchronous Pull**:
    - Issue `workspace/diagnostic` or block on `publishDiagnostics` proxy (See Section 4).
7.  **Finalization**:
    - Send `textDocument/didClose` to relinquish virtual control.
8.  **Knowledge Base Update**:
    - **Synchronous RAG Re-indexing**: Generate and store new embeddings for the file symbols (See Section 5.1).
9.  **Response Construction**:
    - Construct the **Markdown Full Disclosure** report (See Section 7).


### 3.2 `create_file` Lifecycle
- Identical to `edit_file`, but must fire `workspace/didChangeWatchedFiles` immediately after the write to ensure the LSP recognizes the new file/package before the diagnostic pull.

## 4. Deep Dive: Synchronous Diagnostic Capture

To satisfy the "Turn-Level Certainty" requirement, Neko must definitively wait for the LSP to finish its post-save analysis.

### 4.1 The Proxy Mechanism
- **The Challenge**: Most LSPs push diagnostics asynchronously.
- **The Solution**: The `lsp.Client` implements a `DiagnosticWatch` map (`uri -> chan struct{}`).
- **The Algorithm**:
    1.  Tool fires `didSave`.
    2.  Tool registers a channel for the URI.
    3.  `lsp.Client.readLoop` intercepts `textDocument/publishDiagnostics`.
    4.  It updates the global diagnostic cache and signals the channel.
    5.  **Deterministic Block**: The tool waits on the channel with a **2.0s Hard Timeout**.
    6.  On timeout, return the *current* cache (better a slightly stale truth than a hung session).

### 4.2 Non-LSP Degradation (Plugins)
For backends without an LSP:
- Preserve `backend.Validate` (Syntax) and `backend.Format` checks on the individual file.
- Return success with a warning: "LSP unavailable. Run `build` for full verification."

## 5. Pillar 2: RAG-Enhanced Context & Search

### 5.1 Semantic Search (`semantic_search`)
- **Engine**: `chromem-go` (Zero-dependency, local-first vector store).
- **Persistent Storage**: `.neko/embeddings.db`.
- **Maintenance**: Incremental re-index of a file **must be performed synchronously** immediately following every successful `didSave`. This ensures the agent's next turn operates on a search index that reflects the latest edits.
- **Latency Budget**: 1.0s for embedding generation. If using a remote provider (Gemini), the tool blocks until the update is confirmed.

### 5.2 Symbol-Aware Chunking
Standard token-window chunking is rejected. Neko uses the LSP to preserve code meaning:
1.  **Discovery**: Call `documentSymbol` internally to find boundaries. (Note: The user-facing `outline` parameter is deprecated, but the underlying capability is core to RAG and Annotations).
2.  **Assembly**: Each chunk = `[Context Header] + [Symbol Body]`.
3.  **Header Metadata**: Includes File Path, Package Name, and summarized Imports.
- **Result**: Semantic search returns complete functions or classes, providing the agent with executable context.

### 5.3 Type Info (Unified Semantic Engine)
To eliminate "Turn-Waste" for type-checking without degrading code readability, `read_file` output is augmented with a "Type Info" section. This section is powered by the same logic as the `describe` tool, ensuring consistency across the platform.

- **Philosophy**: Keep the code clean. No injected tags or banners. The code block must match the disk exactly. 
- **Mechanism**: Neko iterates over the requested lines, identifies significant symbols (implicit variables `:=`, struct usages, etc.), and invokes the **Enhanced Describe Backend**.
- **Enhanced Describe Backend**:
    - **Depth Limit**: Resolution is strictly limited to **one level deep**. If a symbol is a variable, Neko resolves its type and fetches the hover information for that *type definition*. It must not recurse further into the fields of that type.
    - **Method Set Extraction**: Neko explicitly extracts the full method set and struct fields from the LSP response.
    - **Doc Extraction**: Captures the primary documentation paragraph.

- **Structure & Format (Dense Log)**:
    ```markdown
    ## Type Info
    - Line 10 db (*chromem.DB): NewDB creates a new vector database instance.
    - Line 11 engine (*rag.Engine): Engine manages the local vector database.
      Fields: db, collection, mu, projectRoot
      Methods: IngestFile(...) error, Search(query string, limit int) ([]chromem.Result, error)
    ```

- **Pragmatic Filtering**:
    - **External Scope Only**: Only list type info for symbols that are **NOT** defined in the current file. Repeating local definitions (like a function's own signature) is a waste of context.
    - **Include Local Usage**: Do process local variable assignments (e.g., `db := GetDatabase()`) to reveal the external types they hold.
    - Skip built-in primitives (`string`, `int`, `error`, etc.).
    - Deduplicate entries.

## 6. Detailed Tool Assessment & Strategic Gaps

### 6.1 Refactoring Tools
- **`multi_edit`**: Sequentially applies edits to multiple files.
    - **Failure Policy**: Best-effort. If edit 3 of 5 fails, Neko reports the error but keeps edits 1-2 on disk. The agent is responsible for rollback (via Git) or fixing the broken edit in the next turn.
    - **Validation**: Performs only **one** global diagnostic pull at the end of the batch.
- **`rename_symbol`**: Atomic project-wide rename via LSP `textDocument/rename`. It applies the resulting `WorkspaceEdit` directly to the FS and returns the final project health.

### 6.2 Enriched `find_references`
- **Current Problem**: Flat list of coordinates is uninformative.
- **Upgrade**: Neko will enrich each reference with its containing function name using a `hover` lookup: `- server.go:42 (in 'func establishProject')`.
- **Categorization**: Two pragmatic lists: **[SOURCE]** and **[TESTS]**.

### 6.3 Navigation Upgrades
- **`read_file` Warm-Starting**: Every read triggers `textDocument/didOpen` immediately followed by `textDocument/didClose`. This "warms the cache" and ensures the server indices are current without leaving long-lived virtual sessions open.
- **LSP-Powered Outline**: The `outline` parameter in `read_file` is **preserved** but its implementation is migrated from custom backend parsers to the LSP's `textDocument/documentSymbol`. This provides high-quality, deterministic structural summaries for all LSP-enabled languages.

**JSON is a glitch.** All tools that return project health must use this unified Markdown template:

```markdown
### ✅ [Tool Name] Success: [Context]

[Optional Syntax Snippet with -> Pointer from ExtractErrorSnippet]

🔍 **Current Project Health:**
- `file:line:col`: [Severity] Error Message
- `file:line:col`: [Severity] Error Message

*Total: N errors found across M files.*
```

## 8. Implementation Guide: Low-Level Logic

### 8.1 LSP Protocol Extensions
The Neko LSP client must be updated to support:
- `workspace/diagnostic` (Request/Report)
- `textDocument/formatting`
- `textDocument/codeAction` (with `Only: ["source.organizeImports"]`)
- `textDocument/rename`

### 8.2 Sequential Versioning
- `lsp.Client` must track `openedDocs map[string]int`.
- Every `didOpen` resets the version to 1. 
- Every `didChange` and `didSave` must increment the version by **exactly 1**.

### 8.3 Error Snippet Extraction (Backend-Configured)
- Regex patterns for line extraction must live in the `plugin.json` or native backend configuration, not hardcoded.
- Examples: Go (`:(\d+):(\d+):`), Python (`line (\d+)`).

## 13. Graceful Degradation (LSP-Less Environments)

Neko must remain functional in environments where an LSP server is not installed or supported. The following fallback mechanisms are mandated:

### 13.1 Tiered Intelligence Model
- **Elite Tier (LSP)**: Deep semantic awareness, global diagnostics, and automated save actions.
- **Robust Tier (Native)**: Syntactic awareness, file-level validation, and basic formatting.

### 13.2 Feature Fallback Table
| Feature | LSP Active | Fallback (Native/Plugin) |
| :--- | :--- | :--- |
| **Syntax Guard** | `publishDiagnostics` | `backend.Validate` (Temp File) |
| **Semantic Guard**| `workspace/diagnostic` | **Omit** (Agent relies on `build` tool) |
| **Formatting** | `textDocument/formatting` | `backend.Format` (e.g., `gofmt`, `ruff`) |
| **Outline** | `documentSymbol` | Legacy Backend `Outline()` |
| **RAG Chunking** | Symbol-aware boundaries | Line-based fixed-window chunking |

### 13.3 Reliability Rules
1.  **Silent Fail-Soft**: LSP failures (timeouts, crashes, missing binaries) must never crash an MCP tool call. Neko must log a warning and immediately activate the Robust Tier fallback.
2.  **Capability Awareness**: Tools must query the `Capability` of the active backend. If `CapLSP` is false, the LSP lifecycle is skipped entirely.
3.  **Feedback Honesty**: When operating in Robust Tier, the tool response must include a footer: *Note: LSP unavailable. Global semantic verification skipped.*
